Windows 问题定位与修复计划

原始反馈

1. 新增账号首次登录强制修改密码校验机制
2. 文件删除逻辑优化：删除文件统一移入本地回收站，不做物理直接删除
3. 覆盖安装，需要 kill 掉旧进程再安装，不然会报错
4. 桌面端的消息通知
5. Windows 右下角小图标右键点击经常没反应，需要在任务管理器中杀进程，必须想办法保证它的可用
定位结论

这 5 个问题主要集中在 apps/ai-studio，其中：

- 1 / 4 同时涉及服务端协议与桌面端消费逻辑
- 2 / 5 主要在 Tauri Rust 侧
- 3 主要在 Windows NSIS 安装器
当前代码现状：

- 首次登录强制改密：当前服务端用户模型、登录响应、桌面端登录流均没有“必须修改密码”的状态位，属于功能缺口
- 文件删除进回收站：桌面端 Node Runtime 文件系统删除实现仍直接 remove_file/remove_dir_all，属于行为不符合预期
- 覆盖安装杀旧进程：NSIS 卸载流程会尝试关闭窗口，但安装/升级流程没有可靠的旧进程探测与终止
- 消息通知：消息中心、SSE 未读数、Tauri 系统通知能力都已部分存在，但还不是完整的“新消息提醒 + 系统通知”方案
- 托盘右键无响应：当前托盘配置为左键弹菜单、双击左键打开主窗口，且菜单依赖运行时重建，在 Windows 下稳定性风险较高
关键代码入口：

- 托盘 / 更新 / Tauri 生命周期：
  - apps/ai-studio/src-tauri/src/menu.rs
  - apps/ai-studio/src-tauri/src/lib.rs
- Windows 安装器：
  - apps/ai-studio/src-tauri/nsis/installer.nsi
  - apps/ai-studio/scripts/create-windows-installer.sh
- 登录 / 通知：
  - apps/ai-studio/src/routes/_public/login.lazy.tsx
  - apps/ai-studio/src/hooks/use-notifications.ts
  - apps/ai-studio/src/lib/desktop-notification.ts
- 用户 / 密码：
  - apps/server/src/models/users.ts
  - apps/server/src/services/user/user-password.service.ts
  - apps/server/src/routes/users.ts
实施方案

1. 首次登录强制修改密码

目标：
- 仅针对“管理员分配初始密码”的本地账号，在首次登录或密码被管理员重置后，强制先修改密码再进入工作台
服务端改动：
- 在 users 表新增 must_change_password 布尔字段，默认 false
- 管理员创建本地账号、管理员重置本地账号密码时置为 true
- 用户通过 /me/password 成功自助改密后置回 false
- 登录成功返回中补充 mustChangePassword
桌面端 / Web 端改动：
- 登录成功后读取 mustChangePassword
- 若为 true，进入受限态：
  - 只允许访问修改密码页和退出登录
  - 禁止进入正常工作台
- 复用现有 profile 页面里的改密能力，不新建第二套改密接口
边界规则：
- 仅本地密码账号触发
- LDAP / Google / 手机 OTP 等外部认证账号不触发
- 自注册并自行设置密码的账号不触发
2. 文件删除统一进入本地回收站 / 废纸篓

目标：
- 桌面端删除本地文件时，统一进入系统回收站 / 废纸篓，而不是物理删除
- Windows 必须进入系统回收站
- macOS 必须进入废纸篓
改动范围：
- 优先修改 Tauri Node Runtime 的文件系统删除实现
- 当前 filesystem domain 中 handle_delete 不能再直接 remove_file/remove_dir_all
实现方式：
- Windows 平台删除改为调用系统回收站 API
- macOS 平台删除改为调用系统废纸篓能力
- 删除文件和目录都统一走回收站语义
- 回收站操作失败时直接报错，不静默回退到物理删除
保留物理删除的范围：
- 传输临时文件
- partial 文件
- 缓存文件
- 内部清理逻辑
说明：
- 这次覆盖桌面端本地文件删除，至少包含 Windows 和 macOS
- 服务端 workspace 文件删除、对象存储删除不在本次改动范围内
3. 覆盖安装前自动关闭旧进程

目标：
- Windows 安装器在覆盖安装时自动处理旧进程占用，避免用户手动杀进程
当前问题：
- 卸载逻辑只尝试通过窗口标题关闭程序
- 安装流程本身没有在复制文件前检查和终止 tego-ai-studio.exe
实施方案：
- 在 NSIS 安装主流程开始复制文件前增加“旧进程检测与关闭”步骤
- 优先尝试优雅关闭：
  - 查找主窗口
  - 发送关闭消息
  - 等待短超时
- 若仍然存活：
  - 提示用户
  - 强制终止旧进程
  - 再继续安装
一致性要求：
- 安装和卸载都复用同一套“关闭旧进程”函数
- 自更新安装链路同样适用，避免 updater 下载完成后安装失败
4. 桌面端消息通知增强

目标：
- 补齐“新消息提醒 + 系统通知 + 未读同步”的完整通知体验
当前已有能力：
- 消息中心 Drawer
- 未读数查询
- 通知 SSE
- Tauri 系统通知封装
- Node Runtime overlay 提示
问题：
- 现有通知能力分散，偏消息中心，不是统一通知策略
- 桌面系统通知与应用内提醒没有统一去重和抑制规则
- 当前已经存在通知开关 UI，但桌面通知实际发送链路没有读取该开关，属于“开关已开启/关闭但行为不一致”的现存 bug
- 当前 use-notifications 中收到 SSE 后会直接调用 sendDesktopNotification(...)，没有接入用户偏好里的 notifications.enabled / notifications.desktop
- 因此这项不是从零新增，而是“修复现有部分支持下的通知 bug，并补齐缺失策略”
实施方案：
- 统一通知入口，基于现有通知 SSE 做消息提醒
- 收到新通知时：
  - 刷新未读数
  - 刷新消息中心列表
  - 在桌面端触发系统通知
  - 在应用内展示轻量提醒
- 在真正发送桌面通知前读取用户通知偏好：
  - notifications.enabled = false 时完全不发
  - notifications.enabled = true 且 notifications.desktop = false 时仅保留应用内未读和消息中心刷新，不发系统通知
  - notifications.sound 后续决定是否播放提示音
- 增加去重策略，避免 SSE 重连后重复提醒
- 增加前台抑制策略：
  - 当前已打开消息中心详情或对应上下文时，不重复弹系统通知
- 增加权限状态处理：
  - 用户开启桌面通知开关时，显式检查 / 请求系统通知权限
  - 权限被拒绝时要给出可见反馈，不能静默失败
范围说明：
- 本次优先覆盖消息中心产生的新消息提醒
- node-runtime:notification 现有 overlay 可保留，但需要和消息通知策略区分职责
5. Windows 托盘右键无响应

目标：
- 提升 Windows 托盘菜单可用性，避免必须依赖任务管理器杀进程
- 增加兜底方案，任何情况下都必须对右键有响应
- 确保“退出程序”始终可达且可正常工作
当前风险点：
- 托盘使用 .show_menu_on_left_click(true)，不符合 Windows 常见交互
- 菜单状态变化时通过 set_menu 整体替换，可能导致托盘实例进入异常状态
- 右键没有作为主菜单入口重点保障
- 当前退出逻辑过于依赖托盘菜单入口，一旦托盘异常，用户可能失去正常退出路径
实施方案：
- 调整为 Windows 习惯交互：
  - 右键弹菜单
  - 双击左键打开主窗口
  - 左键不再承担主菜单入口
- 降低高频整菜单重建，优先减少 set_menu 整体替换次数
- 为托盘相关操作增加日志：
  - 托盘创建
  - 菜单重建
  - 点击事件
  - 菜单设置失败
- 增加异常兜底：
  - 菜单设置失败时尝试重建托盘实例
  - 托盘右键事件异常时，立即走降级处理，至少弹出最小兜底菜单
  - 若富菜单构建失败，退化为只包含以下项目的最小菜单：
    - 打开主窗口
    - 打开设置
    - 退出程序
  - 若托盘菜单整体失效，右键动作至少触发主窗口显示，并在主窗口内暴露明显的“退出程序”入口
  - 退出能力不能只依赖托盘入口
硬约束：
- 任何情况下，右键都必须有响应，不能出现“点了完全没反应”
- 任何情况下，用户都必须能在不打开任务管理器的前提下退出程序
- 托盘菜单失败时，允许功能降级，但不允许右键失效和退出失效
退出兜底路径：
- 托盘正常时：托盘菜单内退出
- 托盘菜单降级时：最小兜底菜单内退出
- 托盘事件失效时：自动拉起主窗口，并提供明确的退出按钮 / 菜单项
- 主窗口关闭按钮若当前是“隐藏到托盘”语义，则仍需保留显式“完全退出程序”动作
测试计划

首次登录改密

- 管理员新建本地账号后首次登录，必须先改密
- 改密成功后再次登录，不再触发强制改密
- 管理员重置密码后，用户再次登录重新触发强制改密
- 外部认证账号登录不触发
回收站删除

- Windows 下删除单文件进入回收站
- Windows 下删除目录进入回收站
- macOS 下删除单文件进入废纸篓
- macOS 下删除目录进入废纸篓
- 回收站失败时返回明确错误
- 传输临时文件清理仍保持物理删除
覆盖安装

- 应用运行中执行安装，安装器能自动关闭旧进程并完成安装
- 旧进程卡死时能强制终止并继续安装
- 自更新安装链路不再因文件占用失败
消息通知

- 桌面通知开关关闭时，不发送系统通知，但未读数和消息中心仍正常刷新
- 桌面通知开关开启时，收到新通知会正常弹出系统通知
- 系统通知权限被拒绝时，UI 有明确反馈，不是静默无效
- 收到新通知时未读数增加
- 消息中心列表自动刷新
- 桌面系统通知正常弹出
- SSE 重连后不会重复提醒同一条消息
托盘

- Windows 下右键稳定弹菜单
- 双击左键可打开主窗口
- 多次连接状态切换后托盘仍可用
- 托盘异常时可自动恢复，避免只能靠任务管理器结束进程
- 托盘菜单构建失败时能退化为最小兜底菜单
- 模拟托盘异常后，右键至少还能拉起主窗口或展示最小菜单
- 无论托盘处于正常、降级还是异常恢复状态，退出程序都能正常执行
默认实施顺序

建议按下面顺序落地，先解决最确定、最影响可用性的项：

1. Windows 安装器自动关闭旧进程
2. Windows 托盘右键稳定性
3. 本地文件删除进入回收站
4. 首次登录强制修改密码
5. 桌面端消息通知增强
这样可以先修复安装和存活性问题，再补协议和体验问题。
